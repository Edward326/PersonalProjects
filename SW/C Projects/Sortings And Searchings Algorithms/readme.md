# Sortings And Searchings

## Purpose

Three related C exercises exploring sorting and searching algorithms on data read from text
files: an interactive benchmark that compares six sorting algorithms combined with binary search
over a movie database, and two variants of a string-sorting tool that additionally demonstrate
file-based (external) sorting. All three read their input records from a text file passed on the
command line and are structured as small header/implementation libraries plus a `main.c` driver.

## Implementation

### `AutoBenchMark` — incremental search performance comparison
Reads a tab-separated movie database (`filme.txt`: year, length, title, category, awards) into
a `Movies` array (`record.h`/`record.c`), then benchmarks **six sorting algorithms** — binary
insertion sort, selection sort, bubble sort, shaker (cocktail) sort, Shell sort (Knuth gap
sequence), and quicksort (median-of-array pivot start) — each implemented in `sort.c` with
matching comparison/swap counters.

The core interaction, in `search.c`, is an **incremental prefix search**: as the user types a
movie title character by character (read one keypress at a time via a raw-terminal
`getHiddenChar()`, without waiting for Enter), the working copy of the database is re-sorted by
the current prefix after every keystroke, and `binSearchimpl()` (a binary search over the sorted
prefix) locates and displays all titles matching what's been typed so far, highlighting the
matched prefix in color. This is repeated once per sorting algorithm, each on a fresh copy of the
original array (`make_copy()`), while a `Performance` struct accumulates, per algorithm: total
comparisons and swaps during sorting, total comparisons during searching, and total wall-clock
time (`clock()`). `PerformanceDisplay()` then prints a per-algorithm summary and reports which
algorithm scored best on each metric (fastest overall, fewest swaps, fewest sort comparisons,
fewest search comparisons).

Files: `structure.h` (the `Movie`/`Movies` records), `record.h/.c` (loading a `Movies` array from
a file and freeing it), `sort.h/.c` (the six sorting algorithms), `search.h/.c` (the interactive
benchmark driver and binary search), `main.c` (entry point: loads the file passed as
`argv[1]` and calls `binSearch()`).

### `String_BubbleSort` — array-based string sorting and searching
A smaller, self-contained exercise built around a simple `element { int id; char *value; }`
record read from a plain text file (`filereader.h/.c`). It progressively layers several weekly
assignments (visible as commented-out sections in `main.c`, left in place to show the project's
history):

- **Searching** (`filesearch.h/.c`) — linear search and binary search over the records by value.
- **Sorting** (`filesort.h/.c`) — binary insertion sort, bubble sort, shaker sort and quicksort,
  each tracking the number of element transfers/swaps performed.
- **Two-criteria sort** (`bubblesort_2C` in `filesort.c`) — a bubble sort that orders records
  ascending by their first character and descending by their second character.

The active code path in `main.c` sorts the input file with quicksort, writes the result to
`quicksort.txt`, then re-sorts that output with the two-criteria bubble sort and writes it to
`bubblesort_2C.txt`, so both intermediate and final results are inspectable on disk.

### `String_InterclassSorting` — external (file-based) merge sort
Built on the exact same `element`/search/sort foundation as `String_BubbleSort` (identical
`structure.h`, `filesort.h`, `filesearch.h` APIs), this variant adds an **external interclassare
(merge) sort** in `filesortadv.h/.c`, useful for datasets too large to comfortably sort entirely
in memory:

- `interclass3secv()` implements the classic 3-file natural merge sort: the input is first
  written out to `init.txt`, then repeatedly split into two run files (`aux1.txt`/`aux2.txt`) of
  doubling run-length `k` (`read_secv()`), which are merged back together in sorted run order
  (`interclass2secv()`) — the run length doubles on each pass (`ktuplu *= 2`) until it covers the
  whole file, at which point the data is fully sorted.
- `main.c` runs the same quicksort + two-criteria-bubblesort pipeline as `String_BubbleSort`, and
  additionally invokes `interclass3secv()` on the quicksorted array to demonstrate the external
  merge sort on already-sorted input.

## Notes
- All three programs expect exactly one command-line argument: the path to the input text file.
- Files ending in `~` are editor backups and not part of the buildable source; `Values.txt`,
  `Values_R.txt`, `init.txt`, `aux1.txt`, `aux2.txt`, `quicksort.txt` and `bubblesort_2C.txt` are
  sample input/output data files generated by running the `String_*` programs.
- `AutoBenchMark`'s raw-terminal input handling (`termios`, `ICANON`/`ECHO` toggling) and use of
  `sys/ioctl.h` for terminal width make it Linux-terminal-specific.
